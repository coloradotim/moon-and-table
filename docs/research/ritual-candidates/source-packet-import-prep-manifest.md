## Purpose

This manifest makes the approved extraction candidates mechanically importable without re-opening the source PDFs or asking a later import pass to guess at runtime fields. The source packets remain the canonical user-facing prose artifacts; this file is the normalized runtime mapping layer.

## Import Defaults

For every row below unless explicitly stated otherwise:

- `status`: `draft`
- `origin.type`: `source`
- `availability.findable`: `true`
- `availability.directUseEligible`: `true`
- `adaptationPolicy.purposeChange`: `not_allowed`
- `import readiness label`: `approved_for_mechanical_import`

Recommendation eligibility is separated from search/direct-use. A `recommendationEligible` value of `false` below means the Ritual should still import as findable/direct-use, but Choose with me should not select it until the listed missing gate or owner-review decision exists.

## Mapping Rules

- Adult, explicit, sex-forward, kink-adjacent, spirit, target-linked, adversarial, high-capacity, medical-adjacent, and group labels are preserved as metadata rather than import blockers.
- `yes` and `yes_constrained` recommendation targets become recommendation-eligible when the purpose/carrier/capacity/audience/timing metadata can represent the lane.
- `later_review` becomes recommendation-eligible when the only concern is adult/private/high-capacity fit that can be expressed through current metadata. It remains held only when the current model lacks the necessary gate, such as group audience, target-linked/adversarial work, spirit-working gates, medical/therapy-adjacent gates, or unresolved operative words.
- `no` recommendation targets remain findable/direct-use but recommendation-held.
- `bestWindow`, `whyThisFitsIngredients`, and `howThisWasChosenIngredients` can be generated from this manifest plus the source packet prose.
